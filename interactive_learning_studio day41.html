Classes, Objects & the Meaning of self

A class is a blueprint. It doesn't do anything by itself — it just describes what an object built from it will have (attributes) and be able to do (methods). An object (or instance) is the real thing built from that blueprint, sitting in memory, with its own data.

🏠
Analogy: a house blueprint (the class) specifies "3 bedrooms, 2 bathrooms, a garage." The blueprint never gets cold or leaks a pipe — only an actual house built from it (the object) does. You can build many houses from one blueprint, and each one has its own address, its own furniture, its own occupants — just as each object has its own attribute values.
class Dog:
    # class attribute: shared by every instance
    species = "Canis familiaris"

    def __init__(self, name, breed):
        # instance attributes: unique per object
        self.name = name
        self.breed = breed

    def bark(self):
        return f"{self.name} says woof!"

rex = Dog("Rex", "Labrador")
print(rex.bark())  # Rex says woof!

self is not a keyword — it's just the conventional name for the first parameter of any instance method. Python automatically passes the object itself into that slot whenever you call rex.bark(); it's really shorthand for Dog.bark(rex).

Class attributes vs. instance attributes

Class attributeInstance attribute
Defined directly in the class bodyUsually defined in __init__ via self.x = ...
Shared by every instance (one copy)Belongs to one object only
Good for constants / shared defaultsGood for data that varies per object
Common misconception

Mutable default arguments are dangerous. def __init__(self, items=[]) creates one list object at function-definition time, shared by every instance that doesn't pass its own list. Mutating it in one object silently affects all the others. Always use items=None and then self.items = items if items is not None else [].

Practice exercise

Write a Car class with a class attribute wheels = 4 and instance attributes make and model, plus a method describe() that returns a sentence using both. Sketch it below (not executed, just for practice):

class Car:
    wheels = 4
    def __init__(self, make, model):
        self.make = make
        self.model = model
    def describe(self):
        return f"{self.make} {self.model} has {self.wheels} wheels"
Key takeaways
  • A class is a blueprint; an object is a specific instance built from it.
  • __init__ runs automatically to set up a new object's instance attributes.
  • Class attributes are shared; instance attributes belong to one object.
  • Never use mutable objects as default argument values.

Summary Notes

Object-oriented Python organizes code around classes (blueprints) and objects (instances). The four pillars — encapsulation, inheritance, polymorphism, and abstraction — describe how to protect data, reuse behavior, let different types respond to the same call, and hide unnecessary detail behind clean interfaces. Dunder methods let your objects integrate with Python's own syntax; @property, @classmethod, and @staticmethod refine how attributes and alternate constructors behave. At the advanced end, ABCs, mixins, and Python's MRO let you compose behavior from multiple sources safely, provided you favor composition and shallow hierarchies over deep, tangled inheritance trees.
